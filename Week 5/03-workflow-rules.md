# Git Workflow Rules — [ชื่อทีม]

> ปรับจาก branching workflow ปกติให้ "เร็วพอ" สำหรับ 48 ชม. — ตกลงกติกาให้ครบก่อนเริ่ม แล้วห้ามเปลี่ยนกลางทาง

## Branching Model

- รูปแบบ branch: `main` + `feature/<module>` (อายุสั้นไม่เกิน [2-3] ชม. ต่อกิ่ง	เอาค่าต่ำสุดของช่วงที่เสนอ เพราะเวลารวมมีแค่ ~52 ชม. กิ่งที่อยู่นานกว่านี้เสี่ยง conflict สะสมและงานติดค้างข้ามกันนานเกินไป)
- สิทธิ์ merge เข้า `main` โดยไม่ต้องรอ review: ทุกคน merge เองได้ทันที เงื่อนไขบังคับคือต้อง build ผ่านในเครื่องตัวเองก่อนกันแหก push ทุกครั้งดู Branch Protection เพราะทีมมีแค่ 5 คน แบ่งงานคนละ module อยู่แล้ว การรอ review จริงจะกินเวลาที่ไม่มี และคนอื่นอาจไม่เข้าใจ module ที่ไม่ใช่ของตัวเองพอจะ review ได้แม่นอยู่ดี
- ถ้าไม่มี branch protection แบบเต็มรูปแบบ ใครเป็นคน spot-check ว่า `main` build ผ่านอยู่เสมอ: เลือกคนที่ดูแล core loop/`GameStateManager` เพราะเป็นคนที่รู้ชัดที่สุดว่า build เต็มระบบต้องรันอะไรผ่านบ้าง (ตาม pipeline checklist ตอนนี้คือคนที่ดูแล 126) แต่ต้องเป็นงานที่ทำ "ตลอด" ไม่ใช่ทำครั้งเดียว ทั้งทีมควรยืนยันเองว่าใครจะพอมีเวลาเช็คได้ตลอด

## Commit Convention

| Prefix    | ใช้เมื่อ                                                                                                                                                                                                                                                |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `feat:` | เพิ่ม feature/module ใหม่                                                                                                                                                                                                                              |
| `fix:`  | แก้บั๊ก                                                                                                                                                                                                                                                  |
| `wip:`  | งานที่ยังไม่เสร็จ แต่ต้อง push กันงานหาย                                                                                                                                                                                       |
| asset:    | เพิ่ม/แก้ art, เสียง, ไฟล์ content แยกจาก `feat:` เพราะทีมนี้มีฝั่งอาร์ต 2 คน commit ไฟล์ asset บ่อยกว่าโค้ด การแยก prefix ช่วยให้ scan log หา commit โค้ดง่ายขึ้น |

- กติกา push: push อย่างน้อยทุก [1] ชั่วโมง แม้โค้ดยังไม่เสร็จ (ใช้ `wip:`) เพื่อกันงานหาย
- Pull ก่อนเริ่มงานทุกครั้ง


## Branch Protection ขั้นต่ำ

* `main` ต้อง build ผ่านเสมอ เพราะอาจต้อง submit จาก `main` ได้ทุกเมื่อ
* วิธีการันตี: **"ทดสอบรันก่อน push ทุกครั้ง" (บังคับ)** สำหรับทีม 5 คนในเวลา ~52 ชม. ไม่คุ้มตั้ง CI/GitHub Action เต็มรูปแบบ ยกเว้นมีคนตั้งเสร็จได้ใน 15-20 นาทีแรกโดยไม่กระทบงานหลักของตัวเอง ถ้าไม่มั่นใจให้  self-test ก่อน push เป็นการบังคับแทน

## Merge Conflict — เมื่อเกิดขึ้นกลางดึก

1. คนที่เจอ conflict แจ้งใน group chat ทันที พร้อมชื่อไฟล์
2. ถ้าเจ้าของไฟล์ (ดู **02-roles.md**) ออนไลน์ > ให้เจ้าของไฟล์ตัดสินใจ
3. ถ้าเจ้าของไฟล์ไม่อยู่ ผู้ตัดสินใจสำรองคือคนที่ตอนนั้นงานตัวเองไม่ได้ยุ่ง
4. ห้ามแก้ conflict แบบเดาเอง ถ้าไม่แน่ใจให้รอคนตัดสินใจหลักของทีม ถึงจะต้องรอ 10-15 นาที
